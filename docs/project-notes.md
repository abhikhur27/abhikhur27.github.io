## Portfolio Listing Policy (Hiring + YC Bar)

Only list projects on the homepage when they are fully built out and clearly defensible to:

- a hiring manager evaluating technical depth + product judgment
- an early-stage investor evaluating founder execution quality

Do not list random tools, thin prototypes, or one-file utilities unless they are polished into a complete product story.

## Publish Gate Checklist

A project must pass all items before homepage inclusion:

1. Problem quality: solves a real pain point with clear target user and frequency.
2. Product completeness: has complete workflow, sharp UX, and no "demo-only" dead ends.
3. Technical depth: non-trivial implementation decisions are documented and testable.
4. Reliability: predictable behavior on valid/invalid inputs with error handling.
5. Proof of quality: sanity checks/tests and reproducible usage examples exist.
6. Credibility: README and demo can survive technical interview follow-up questions.
7. Distinctiveness: clearly different from existing portfolio projects by mechanism and outcome.

## Incubation Rule

If a project fails any gate, keep it off the homepage and track progress in a dedicated development-cycle markdown file until it is ready.
